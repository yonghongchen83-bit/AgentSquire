# Decisions

## Q6 real UX: a persisted chat message, not a richer toast

The interim UX (from `squire-adapter`) emitted a `stream-error` string plus an `output:append` diagnostic log line on `TurnOutcome::Failed`. Neither is durable — both are transient IPC events with no row in `ConversationStore`, so refreshing the conversation or reopening the app loses the evidence entirely. Q6's stated UX intent ("user can inspect the failed response and adjust next prompt/direction") requires the failed content to still be inspectable after the moment it happened, which a toast can't provide.

Decision: `SquireContextAdapter::finalize_turn`, on reaching `TurnOutcome::Failed`, now calls `ConversationStore::append_message` itself (via the new `reject_and_record` helper) with a formatted assistant-role message containing the reason and the full failed content, before returning the outcome to orchestration. This reuses the exact mechanism `LegacyContextAdapter::finalize_turn` and the compliant-response path in `SquireContextAdapter::finalize_turn` already use to persist assistant turns — no new persistence primitive needed, no frontend changes needed. `streaming_cmd.rs`'s existing `onStreamError` handler already calls `getConversation` and reloads `messages` on any `stream-error`, so the persisted failure message shows up in the chat pane automatically the moment the existing frontend error-handling path runs.

Rejected alternative: adding a distinct `Block`/message-type in the frontend (`src/types/ipc.ts`) for compliance failures, styled differently from a normal assistant message. This would be a nicer visual treatment (e.g. a red-bordered card) but is materially more frontend work (new `Block` variant, `chat-blocks.tsx` rendering branch, `block-parser.ts` handling) for a node whose sibling nodes (`squire-adapter`, `squire-storage`) were both backend-only sessions. The plain-markdown assistant message (using `**bold**` + a `---` separator, consistent with markdown already rendered elsewhere in chat) satisfies Q6's literal requirement (both reason and content are shown) without a frontend change; a follow-up node can add a distinguishing visual treatment later without needing to touch the backend persistence logic again.

`streaming_cmd.rs` still emits `stream-error` (so the frontend's generic error banner fires, telling the user *that* something failed) and, under `verbose_logging`, an `output:append` diagnostic — but the diagnostic is no longer the primary mechanism carrying the failure content to the user, just a debug-log echo consistent with how other verbose-logging blocks in this file work. This is a smaller change than fully replacing `stream-error` — kept because the frontend has no other "turn ended abnormally, stop the spinner" signal today, and building one felt like scope creep for what Q6 actually asked for.

## Structured failure metadata: new `SquireStore` method + table, not reuse of `squire_relationships`

Q6 explicitly enumerates the fields to persist: rule violated, validator reason, retry count, timestamp. None of the four existing `squire-storage` tables fit this shape (tokens/relationships/turns/preserve-lists are all protocol-state tables, not diagnostic logs). Added `SquireStore::record_compliance_failure(ComplianceFailureRecord)` (append-only, write-only from the adapter's perspective — nothing in this node reads it back to drive runtime behavior, matching the "for debugging" framing in Q6) and a fifth LanceDB table, `squire_compliance_failures`, holding `session_id`/`rule`/`reason`/`retry_count`/`failed_content`/`timestamp` as plain string/uint64 columns (timestamp stored as RFC3339 text, consistent with this module's existing preference for simple string columns over Arrow's native timestamp type elsewhere — see `squire-storage/decisions.md`'s general schema notes).

`rule` vs `reason`: the existing rejection paths (`validate_squire_response`'s spec-table checks, `finalize_turn`'s malformed-JSON and ask_user-gap checks) all produce a single free-text `reason` string, not a separate machine-stable rule identifier — the spec (§8.3) is itself just a table of `Condition -> Rejection reason` string literals, no rule-id column. Rather than plumbing a new enum through `validate_squire_response`/`ComplianceFailure` (a `squire-adapter`-owned type this node didn't want to widen unnecessarily), added `classify_rejection_rule(reason: &str) -> String` in `squire.rs`: a small string-matching classifier run only at the point of persistence (`reject_and_record`), producing stable short ids (`malformed_json`, `ask_user_content_conflict`, `empty_close_response`, `undisplayable_token`, `unclosed_span`, `non_invocable_token`, `other`). This keeps the classification logic in one place, next to where it's consumed, and degrades gracefully (`"other"`) if reason wording ever changes upstream instead of silently mis-tagging or panicking.

## `reject_and_record`: a wrapper around `reject`, not a rewrite

`SquireContextAdapter::reject` (existing, from `squire-adapter`) is a sync fn — it has no access to `self.store` (would need to be async) and no `session_id`/`ConversationStore` parameter (both needed to persist the failure). Rather than changing `reject`'s signature (touching all three of its call sites just to thread through two params that are only needed on the terminal-failure branch, one of three possible outcomes), added `reject_and_record` as a thin async wrapper: calls `reject` for the existing retry/fail decision and message-list bookkeeping unchanged, then — only when the outcome is `Failed` — does the two new side effects (record + persist visible message) before returning. All three `finalize_turn` call sites that used to call `self.reject(...)` directly now call `self.reject_and_record(session_id, ..., store).await` instead; `reject` itself is unchanged and still covered by its original test coverage path indirectly through `reject_and_record`.

## Preserve-list lifecycle (Q7): startup-time clear, not per-turn or per-session-close

Q7's rule is "preserve list applies only to the immediate next turn; restart clears pending preserve carryover state." The "next turn only" half was already correctly implemented by `squire-storage` (`set_preserve_list` deletes-then-inserts wholesale every turn close — no accumulation, confirmed by `squire-storage/decisions.md`). The missing half was "restart clears carryover": since `squire_preserve_lists` is now a durable LanceDB table (post-`squire-storage`), a preserve-list written at the end of session A's last turn before an app restart would otherwise still be sitting there, ready to be read back by `build_turn_input` on session A's *next* turn after restart — silently violating "ephemeral handoff, not long-lived continuity state."

Decision: added `SquireStore::clear_all_preserve_lists()` (unconditional — clears every session's preserve-list rows, not scoped to one session, since Q7's language is about the *mechanism* being ephemeral state per se, not about any one session's continuity) and call it exactly once, synchronously, in `setup_cmd.rs`'s `setup_app_impl`, immediately after the `LanceDbSquireStore` is opened and before `app.manage(AppState {...})` runs — i.e. before any session can possibly call `build_turn_input`/`preserved_tokens`. This is a "clear at boot" design, not "clear at session close" or "expire after N turns": Q7's own wording ("restart clears... state") names the *app process boundary* as the clearing trigger, not any per-session or time-based expiry, so boot-time-once is the literal, minimal-surface-area implementation of the stated rule.

`InMemorySquireStore::clear_all_preserve_lists` also implements the method (trivially, `preserve_lists.lock().await.clear()`) for trait completeness and test-double parity, even though its state is already wiped on every process restart by construction (non-persistent) — the method still has an observable effect mid-process (clearing without restarting), which is exercised by its own unit test for defensiveness/documentation value, not because anything in this node's production code calls it mid-session.

## Two flagged follow-ups considered for this node, deliberately not absorbed

`squire-adapter/todo.json`'s sa-4 (raw stream leaks protocol sigils before `finalize_turn` expands them) was explicitly named in that todo item as "closest fit is rejection-ux's 'compliance-failure visibility' scope." On inspection, sa-4 is about the *live-streaming display boundary* for every turn (compliant or not) — it's a `stream-chunk` buffering change in `streaming_cmd.rs`'s tool-loop/generation path, unrelated to what happens after `finalize_turn` returns an outcome. It doesn't touch `TurnOutcome::Failed`, preserve-lists, or failure metadata at all. Judged out of fit and left deferred (`ru-7`) rather than stretched to justify inclusion.

sa-5 (the `ask_user` response-field loop, spec §9.3) would require new IPC (a question-surfacing command), new frontend UI (a way to collect the answer and route it back), and a resubmission-with-accumulated-`user_request` loop in `finalize_turn` — a materially larger feature than "give the user visibility into an outcome that already exists," which is this node's actual scope. Deferred (`ru-8`).

`squire-storage/todo.json`'s ss-9 (real tool-token ingestion) has no relationship to rejection UX or preserve-list lifecycle at all — not reconsidered here, left exactly where it was.
