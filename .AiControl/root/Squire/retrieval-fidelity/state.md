## Closure summary

Both spec-flagged core-differentiator gaps are implemented and tested. **Graph traversal**: `explore()`'s `num_hops` parameter now performs a real undirected BFS over the `squire_relationships` triplet store in both `InMemorySquireStore` and `LanceDbSquireStore`, via a shared `traverse_relationships` helper — tokens discovered purely through graph connection (not matching the query directly) are now returned, each carrying `hop_distance` and `via_token_id` provenance, with a hop-distance-decayed score (`base_score * 0.5^hop`) and `max_results` still applied as a final cap on the combined result set. **Scoring**: `accumulated_hits: u64` is now a real field on token records in both backends, incremented at `upsert_token` (every call, per spec §9.4 step 5 / §5.2), `preserved_tokens` (per spec §3.3's preserve-list-load event), and via a new `record_hit` method called from `token_to_detail`'s tool execution (per spec §6.2, also standing in for §5.1's "chunk loaded into context" event for the store-backed path). `effective_priority` is computed with the spec's exact formula and used as a near-tie secondary sort key in both stores' `explore_memory` ranking.

Deliberate, documented scope narrowing: 2 of the spec's 4 listed hit-count-increment events are wired directly; the other 2 ("explore results acted on" and "§! reference in a loaded chunk") are covered by a narrower `token_to_detail`-call-based proxy rather than a full context-composition scan, since no such scan exists anywhere in this codebase's turn-assembly flow (see decisions.md's full reasoning). This is tracked as `rf-13`, a deferred follow-up, not silently treated as complete.

`ss-9` (real tool-token ingestion) was checked for entanglement and found genuinely separate — traversal and scoring both operate on whatever tokens already exist regardless of type, so a future ingestion feature inherits both automatically once it starts writing real store rows.
