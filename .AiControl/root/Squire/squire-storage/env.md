- Parent node: root/Squire
- Node path: root/Squire/squire-storage
- Objective: Implement the Squire storage stack on LanceDB from day one (Q4) — structured and raw partitions with production-parity triplet store and vector search.
- Scope now: LanceDB schema for structured/raw partitions, triplet store + vector search path, storage interface abstraction for test doubles.
- Non-goal now: adapter control flow (see squire-adapter), UX/diagnostics (see rejection-ux), real tool-token ingestion (mapping discovered MCP/local tools into persisted `SquireStore` tokens with invocable endpoints — `explore(resource_type="tool"/"tool_skill")` still reads live from `ToolRegistry`, not the store; see decisions.md).

- Status: implemented and landed 2026-07-02.

## Durable facts (added this session)

- `LanceDbSquireStore` (`src-tauri/src/storage/squire_lancedb.rs`) is the real `SquireStore` impl. `AppState.squire_store` in `setup_cmd.rs` constructs it at `<app_config_dir>/squire_lancedb` (one LanceDB directory holding 4 tables: `squire_tokens`, `squire_relationships`, `squire_turns`, `squire_preserve_lists`). State now persists across app restarts (previously in-memory only).
- Build prerequisite: the `lancedb` crate's transitive dependency `lance-encoding` requires the system `protoc` (protobuf compiler) binary at build-script time for **cold** builds only (first build after `cargo clean`, a fresh checkout, or a `Cargo.lock` change touching `lance-encoding`). Once compiled, `protoc` is not needed again until the next cold build. Setup: `winget install --id Google.Protobuf -e`, then restart your shell/IDE so the updated user `PATH` takes effect (winget appends `...\WinGet\Packages\Google.Protobuf_Microsoft.Winget.Source_8wekyb3d8bbwe\bin` to the user PATH; it does not apply to already-running shells). If `protoc` still isn't found after restarting, set `PROTOC=<path-to-protoc.exe>` explicitly for the `cargo build`/`cargo test` invocation.
- `arrow-array`/`arrow-schema` must be pinned to `53.2` (matching `lancedb 0.14.1`'s own direct arrow dependency) in `Cargo.toml`, not the `52.x` that `lance` (a transitive dep of `lancedb`) uses internally — the two coexist in the dependency graph (cargo allows multiple versions of the same crate), but our own code must import the 53.2 line to match the types `lancedb::Table`/`Query` methods expose publicly, or you get "no method found" / "multiple different versions of crate" errors.
- Vector search in `explore_memory` is powered by a deterministic hash-based bag-of-words embedding (`embed_text`, 64-dim, FNV hashing), not a real embedding model — no embedding-model provider exists in this codebase yet. This is a documented, swappable placeholder: changing `embed_text`'s body is the only change needed to plug in a real model later (schema, trait, and callers are unaffected as long as the new embedding is also 64-dim or the column is migrated).
- `SquireInvokeTool` now holds both `tool_registry` (primary, authoritative source of real invocable tools) and `store` (fallback via `store.token_detail`). The store fallback exists for forward-compatibility with a future tool-token-ingestion feature (not built in this node) — today it will only ever produce a "no invocable endpoint bound yet" error for tokens the model discovered but that were never registered as real tools, since nothing currently writes tool tokens into `SquireStore`.
