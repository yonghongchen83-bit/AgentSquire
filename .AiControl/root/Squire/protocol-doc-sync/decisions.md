# Decisions

## Approach: inline "Implementation status (runtime v1)" annotations, not a rewrite

Considered rewriting `context_squire_spec_v2.md` wholesale to describe only what runtime v1 does, dropping everything else. Rejected: the spec is the design-time reference for the *full* protocol vision (including deferred/future work like graph traversal and hit-count decay), and several sibling nodes' decisions.md files already cite specific spec sections as their contract (e.g. `squire-adapter/decisions.md`'s "Validation gates implemented (Q6, spec §8.3)"). Silently trimming those sections would break that cross-referencing and erase the design intent for future nodes that might actually implement e.g. graph traversal or hit-count scoring.

Decision: keep the full v2 document intact and add inline "**Implementation status (runtime v1):**" notes at every point where behavior is confirmed accurate, confirmed different, or confirmed unimplemented. This keeps the spec usable both as "what was designed" and "what exists today" without needing two separate documents, and makes future drift-checking easier (grep for the marker string). A top-of-document banner (added to `context_squire_spec_v2.md`) orients a first-time reader before they hit the first inline note.

## Full divergence inventory and per-item judgment

Systematically diffed both docs against `squire.rs`/`squire_lancedb.rs`/`context_adapter.rs`/`streaming_cmd.rs`/`setup_cmd.rs`, cross-checked against `planning/decisions.md` Q1–Q7 and all three implementation nodes' decisions.md files to separate *intentional* decisions from *unintentional* drift. Fourteen divergence points were found (the two explicitly flagged in the task, plus twelve more found by the systematic pass). Each is resolved below as either **(A) doc updated to describe actual, intended runtime behavior** or **(B) flagged as a genuine unimplemented gap** — never silently normalized as if it were finished.

### (A) Doc updated — describes intentional/judged runtime behavior

1. **§8.3 / §14 stale "closes the turn without storing a response" line (the task's flagged item).** Q6's Option 3 was fully implemented by `rejection-ux`: on final failure, `reject_and_record` persists a visible assistant-role chat message (reason + failed content) via `ConversationStore::append_message`, plus a structured `ComplianceFailureRecord` via `SquireStore::record_compliance_failure`. This is strictly *more* than the original spec asked for on this point, not a shortfall — updated §8.3 and §14 to describe the real behavior, and added the two adapter-level rejection reasons (malformed JSON, unwired-ask_user) that exist in runtime but were never in the spec's original 5-row table.

2. **`SQUIRE_SYSTEM_PROMPT` (squire.rs) vs `context_squire_system_prompt_v2.md` (the task's second flagged item).** Per `squire-adapter/decisions.md`'s "Protocol adaptation" section, this was always a deliberate rewrite, not an `include_str!` — the runtime transport is provider-native tool-calling, so the system prompt travels as a `ChatMessage::System`, and `build_turn_input` never replays `session.messages` history (the entire point of Squire mode). Added a governance/reconciliation banner at the top of the system-prompt doc explaining both structural reasons, plus a matching transport note under spec §8.1. Confirmed by direct comparison that the *substance* (three built-ins, two sigils, response JSON shape, validity rules, resubmit-on-rejection guidance) is faithfully preserved in the condensed runtime string — this is a phrasing/transport adaptation, not a content gap, so no further doc change needed beyond documenting *why* the two texts differ.

3. **Q1 explore() superset (max_results, resource_type additive types) not yet applied to spec §6.1.** This was `protocol-doc-sync`'s original, pre-existing scope per `env.md`/`prompt.md` (predates this session's two newly-flagged items) — Q1's decision already states "spec document is now lagging and must be updated to match implementation contract." Updated §6.1's signature, parameter table, and `resource_type` enum to the full superset (`tool_skill`/`concept`/`referential` additive to the original five values), matching both the runtime tool schema (`built_in_tool_definitions()`) and the system-prompt doc's version of this contract, which already had the superset.

4. **`explore(resource_type="tool_skill")` return shape.** The system-prompt doc describes a nested `{"tool": [...], "skill": [...]}` dict; `SquireExploreTool::execute` in `squire.rs` actually returns one flat array with a `type` field distinguishing subtypes. Judged this as an intentional-enough implementation choice to document as the real contract rather than a "bug to fix" — the flat shape is simpler for the model to consume (one array to iterate, not two conditionally-present keys) and was clearly a considered choice at implementation time (`SquireExploreTool` explicitly builds `tool_results` then `.extend()`s skill results into the same vec, rather than constructing a two-key object). Updated both docs: spec §6.1 documents the flat shape as the "runtime v1 actual shape" with the original nested dict kept as a labeled historical reference; the system-prompt doc does the same with a struck-through legacy example. Not treated as a bug because nothing about the flat shape breaks the model's ability to use the results — it just needs the `type` field it already receives.

5. **`prefetched_tokens`/`preserved_tokens` entries carry a `type` field in runtime v1**, present in the system-prompt doc's example but not in the spec's §8.1 example. Minor, cosmetic — noted in §8.1's transport note as an "illustrative omission, not a behavioral divergence" rather than given its own heavyweight entry, since `TokenSummary::token_type` has existed since `squire-adapter` and nothing about this needed a decision either way.

6. **§14 CLI framing.** The actual product is the Tauri desktop app, not a terminal REPL. Added a short implementation-status note rather than rewriting the section, since the *behavioral* rules described (display expansion, rejection handling) do transfer to the real chat UI — only the "printed to a terminal" literal framing doesn't. Treated as illustrative-reference-interface framing, not a functional gap.

7. **§11 "Session end: flush all in-memory state to disk."** Not applicable to runtime v1's storage model — `LanceDbSquireStore` writes are immediately durable (each `SquireStore` write method commits to its LanceDB table synchronously within the call), so there is no batched in-memory state requiring an explicit end-of-session flush step, and none was built. Documented as "not applicable" rather than "not implemented," since building a flush step would be solving a problem the storage model doesn't have.

### (B) Flagged as genuine gap — not silently absorbed as spec-accurate

8. **`ask_user` response-field loop (spec §8.2/§9.3, system-prompt doc's `ask_user` section).** `finalize_turn` returns a hard `Err` the instant `ask_user` is populated — no surface-to-UI, no answer-collection, no resubmit-with-accumulated-`user_request`. This is `squire-adapter/todo.json` sa-5, already known and explicitly flagged as a gap in `squire-adapter/decisions.md` ("Known gaps deliberately left open") and `rejection-ux/decisions.md` (considered and deferred). Not a new discovery, but the spec/system-prompt docs never reflected it — both now carry explicit "NOT implemented" notes at every place this loop is described, distinguishing it clearly from the *other* AskUser path (`invoke("TOOL_AskUser", ...)`), which does work today through the ordinary tool-call loop.

9. **Graph traversal (`num_hops`, spec §4.2/§6.1/§7.1/§10.1).** `num_hops` is accepted end-to-end (tool schema → `SquireStore::explore_memory` signature) by both `InMemorySquireStore` and `LanceDbSquireStore`, but neither implementation performs any triplet-store traversal — `explore_memory` in both stores is a flat type + similarity filter, full stop, regardless of the `num_hops` value. The triplet store itself is written to (`insert_relationship` is implemented and tested in both backends) but nothing reads it back for expansion. This was flagged as a known limitation in `squire-adapter/decisions.md` ("`InMemorySquireStore`... does no real graph traversal — it's a flat filter") for the in-memory stand-in, but `squire-storage/decisions.md` never explicitly re-flagged that the *production* LanceDB implementation also skipped it — reading `squire_lancedb.rs`'s `explore_memory` confirms it inherited the same flat-filter limitation rather than adding real traversal. Judged as a genuine, still-unclaimed gap (not assigned to any node's todo.json) rather than an intentional simplification — flagged at every relevant spec section rather than left to look like a finished Layer-2 retrieval story.

10. **`accumulated_hits`/`effective_priority` scoring (spec §3.2/§3.3/§5.1/§5.2/§6.1/§6.2/§9.4/§10.1, glossary).** No token record in the runtime (`TokenSummary`, `TokenDetail`, or the LanceDB `squire_tokens` schema) has an `accumulated_hits` field; `effective_priority` is never computed anywhere. `creation_turn` is stored but unused for ranking. Ranking is cosine-similarity-over-a-placeholder-embedding plus a substring-match boost (an intentional, separately-documented choice in `squire-storage/decisions.md`'s "Embedding function" section) — but that decision was about *what powers similarity scoring*, not about *replacing the hit-count/decay model entirely*, and nothing in any decisions.md explicitly decided to drop hit-count tracking. Judged as unintentional scope narrowing during implementation (the storage/adapter nodes built what was needed to make `explore_memory`/tool boundary/validation work, and the hit-count bookkeeping — which nothing else in the built system depends on functionally — was never separately picked up) rather than a deliberate decision. Flagged at every affected section rather than silently dropped from the reader's expectations, with a legend entry in the glossary (§17) for quick scanning.

11. **User-input auto-chunking into `USR_TN_NNN` tokens (spec §3.1/§4.3/§9.1 step 2, §11).** `build_turn_input` takes the latest user message's raw text directly as `user_request` — no chunking, no token creation. Same judgment as #10: never explicitly descoped in planning, emerged as a gap because nothing downstream (validation, tool boundary, storage schema) required it to exist for the shipped features to work. Flagged, not silently treated as done.

12. **Raw-partition audit-log storage (spec §4.1/§4.3/§9.4 step 4/§11, glossary).** `squire_lancedb.rs` has five tables (`squire_tokens`, `squire_relationships`, `squire_turns`, `squire_preserve_lists`, `squire_compliance_failures`) — none hold unmarked/raw AI output as a distinct audit log. Unmarked content is not persisted as memory at all beyond the ordinary chat-history message `finalize_turn` already writes via `ConversationStore`. Same category as #10/#11 — flagged, not silently dropped.

13. **§10.1 bootstrap construction mismatches beyond traversal:** trim-by-count (`max_results`, default 10) instead of trim-by-character-budget (`bootstrap_token_limit ÷ 4`); no edge-count tiebreak (no traversal to derive edge counts from); `bootstrap_top_n` from §15's configuration table isn't wired to anything (the "top-N before trim" value is hardcoded to 10 inside `build_turn_input`'s `explore_memory` call, not read from a config value). None of these were separately called out anywhere before this pass. Flagged inline per-step in §10.1 rather than treated as accurate.

14. **§8.3's original 5-row rejection table was incomplete relative to runtime's actual rejection surface** (missing the malformed-JSON and unwired-ask_user cases) — folded into item #1's fix above rather than treated as a separate item, since it's part of the same section's stale-content problem.

## Why items 9–13 are judged "drift," not "descoped," despite no node claiming them yet

Checked every planning decision (Q1–Q7) and every implementation node's `env.md`/`decisions.md` non-goals section for an explicit statement descoping traversal, hit-counts, auto-chunking, or raw-partition storage. None exists — `squire-adapter/env.md`'s non-goals are "storage internals" and "rejection UX," `squire-storage/env.md`'s non-goals are "adapter control flow," "UX/diagnostics," and "real tool-token ingestion" (ss-9), `rejection-ux/env.md`'s non-goals are "protocol validation logic," "ask_user UI round-trip" (sa-5), "live-stream sigil-leak buffering" (sa-4), and "real tool-token ingestion" (ss-9, restated). None of the three nodes' scope statements mention traversal, hit-counts, auto-chunking, or raw-partition storage as either in-scope or explicitly deferred — they simply never came up, because nothing else in the built feature set exercised them. This is the textbook shape of unintentional scope narrowing during incremental delivery (each node built exactly what its own acceptance bar required), not a considered decision to cut these features. Per the task's explicit instruction not to "silently paper over real bugs as spec updates," these five items (9–13) are documented as open gaps in this node's closure rather than folded into a doc-only "this is now the contract" framing the way items 1–7 were.

None of items 9–13 are claimed by any todo.json as of this node's completion — they are new findings from this pass, not previously-known-and-deferred work like sa-4/sa-5/ss-9. Whether to open new follow-up nodes for them is left to `root/Squire/state.md`'s epic-level judgment call (see this node's `state.md` closure summary and `root/Squire/handoff.md`).
